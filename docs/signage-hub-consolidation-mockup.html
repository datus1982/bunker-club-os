WHAT MAKES THIS REAL — the one structural change under the hood
Today a signage item belongs to one screen (signage_items.slot_id). The library model needs items to be venue-wide and a small junction table to say "this asset runs on that screen, in this position, for this long." Nothing on the TVs changes at cutover.
-- TODAY: an item is nailed to exactly one screen signage_items ( id, slot_id, template, fields jsonb, sort_order, duration_seconds, active, … ) -- PROPOSED: the item becomes a venue-wide ASSET (drop the slot binding)… signage_items ( id, venue_id, template, fields jsonb, active, … ) -- no slot_id -- …and a junction says which screens it's queued on, where, and for how long CREATE TABLE slot_queue ( slot_id references signage_slots, item_id references signage_items, position int, -- per-screen order (was signage_items.sort_order) duration_seconds int, -- per-screen dwell (the "different queues" ask) active boolean, primary key (slot_id, item_id) ); -- per-screen takeover: a nullable slot scopes the broadcast (null = all screens) ALTER TABLE screen_takeovers ADD COLUMN slot_id uuid null references signage_slots;
1Migration. New slot_queue junction + move position/duration_seconds/active onto it; drop signage_items.slot_id. Add nullable screen_takeovers.slot_id. RLS mirrors the existing has_module('signage') pattern.
2Backfill (nothing visible changes). Each existing item becomes an asset queued on its original screen — one slot_queue row per item carrying its old sort_order/duration_seconds. The TVs render byte-identically at cutover.
3Reader. resolveRotation() already interleaves authored items + event cards + ★ trailers. It now reads a screen's assets via slot_queue instead of signage_items.slot_id — same output shape, so the public SlotDisplay and the hub/TV parity invariant hold.
4Live-queue editor = the QUEUE surface. The PR #29 EDIT ROTATION editor becomes the slide-over, with library assets in place of slot-bound items (its ▲/▼/SECS now write slot_queue.position/duration_seconds). Events stay venue-wide and materialize in exactly as they do today.
5Editors move to slide-overs. ItemEditor (asset create/edit + DELETE), the event editor, and the takeover console all open over the hub — no routed sub-pages. /signage/events and /signage/broadcast routes retire (redirect to /signage).
Honest cost callout. The migration itself is routine, but slot_id is load-bearing across the whole signage module: ItemEditor writes it, SignageHub groups items by it (itemsBySlot), EditRotation filters by it, resolveRotation's callers pass slot-scoped lists, and the ★-materialization + preview paths all assume one-item-one-screen. This is a real refactor of the module's data spine, not a UI reskin — plan it as its own phase with a reviewer pass, exactly like PR #29. Events being already venue-wide is the free part; the item→asset flip is the work.
Everything the owner asked for — one-click add, shared assets / different queues, cleaner cards, retired Events & Broadcast tabs, all-in-the-hub — is reachable with this one junction. No change to Toast sync, the display engine's render, or the trivia takeover ladder.
