WHY: A schedule flips programs by daypart. It can be computed two ways: on the screen itself every render (like the events stage engine, no new server plumbing) or a server tick that writes the program (like the events cron). One is dramatically more outage-proof.
OPTION A
CLIENT-DERIVED every render
Each TV reads the schedule rows and derives "what program is active right now" itself — no cron, no new table writes. Exactly how event stages already work (eventStage()).
Outage-proof: a TV that boots at 5 PM immediately shows the right daypart. Every screen flips independently at the boundary — a dropped realtime connection can't miss a flip.
Failure mode: a TV with a badly wrong clock flips at the wrong minute. Mitigated — TVs are NTP-synced; the boundary is venue-local wall time from Intl.
A · CLIENT-DERIVED — recommended
OPTION B
CRON TICK writes program
A pg_cron job (1/min) writes the active program into signage_slots.program; TVs pick it up via realtime — like tick_scheduled_events().
Failure mode: a TV offline at the boundary misses the realtime write and stays on the old program until the next change. And the write collides with a manual override in the same column — you lose the "is this manual or scheduled?" distinction (breaks D4).
B · CRON TICK
What stays: either way, resolveSlotMode is untouched — a program still only renders at the bottom of the ladder, so takeover / MOMENT / live game preempt a scheduled program exactly as they preempt rotation today.
D3RECOMMENDED: A — client-derived. A pure scheduleResolve.ts (no react/supabase, like eventStage.ts) computes the active program + the next boundary from anon-readable slot_program_schedule rows, re-derived on the existing 30s tick plus a precise timeout to the next boundary. No cron, no new writes, and it keeps manual-vs-scheduled distinct (D4). Alt B rejected: a cron write is less resilient than client derivation (offline TVs miss it) and would overwrite the manual-override column.

When a schedule is active and you flip a program by hand, it becomes an OVERRIDE that holds until the current daypart's next boundary — then the schedule quietly reclaims the screen. A RESUME SCHEDULE ↺ button hands it back immediately.
Today's behavior is preserved exactly. When NO schedule covers a screen, a hand flip is a permanent pin (program_until = null) — identical to M1/M2. Schedules only change what a flip means while a daypart is running.
This amends the one-line "schedule > manual" in docs/15 §Schedules, which was written before this tension was thought through. The bar-correct rule is manual-wins-until-boundary.
D4RECOMMENDED: a manual (or Q-SYS) program flip during an active daypart sets an OVERRIDE that wins until the next schedule boundary, then auto-resumes the schedule; a RESUME SCHEDULE control clears it at once. With no schedule present, a flip is a permanent pin (unchanged from today). Stored as signage_slots.program + a nullable program_until. Alts: (b) manual pins permanently until a FOLLOW-SCHEDULE toggle is pressed — more surprising ("why is the game still on Monday morning?"); (c) manual only wins for the current minute — too fragile. Reply if you'd prefer (b).

Multiview is always framed and never suppresses chrome. The main region keeps its header + ticker; the panel is a whole extra promo surface. It's the opposite of full-bleed — the maximal-ads program, which is the entire reason it exists.
A "full-bleed main inside multiview" (drop the main's header/ticker, keep only the panel) was considered and rejected: the panel already guarantees ads, and dropping the header/ticker buys only the 171px letterbox bands — which the ratio-locked 16:9 stage can't grow into anyway. No upside.
D7RECOMMENDED: multiview does NOT count as full-bleed — it is always framed. All chrome survives (header + ticker on the main region) PLUS the promo panel. There is no full-bleed multiview toggle. Full-bleed stays a plain-playlist choice (M1 mockup D6, ratified) for movies where ads aren't wanted.

Whole-multiview preemption is automatic. Multiview lives at the bottom of the ladder like every program. When mode flips off 'rotation' (takeover / MOMENT / game), the ENTIRE multiview — main video AND panel — unmounts; the standard full-1920×1080 surface takes over. The <video> stops, the capture MediaStream tracks stop, both with it. When the interruption ends, multiview resumes.
Active WINDOW/MESSAGE promos still surface — in the panel's rotation (they're venue-wide and the panel runs the full portrait resolver), not layered over the main video. The main region stays the video.
No resolver change: resolveSlotMode is untouched — the same one source of truth the hub chip reads. Preemption needs no new code, just the multiview renderer sitting where playlist sits today.
D9RECOMMENDED: multiview is a program at the bottom of the mode ladder; takeover / MOMENT / live game preempt the whole multiview (main + panel) exactly as they preempt rotation today. resolveSlotMode stays unchanged.

Dayparts stack: whichever covers "now" runs; if two overlap, the lower one in the list wins (an explicit tiebreak). Any time no daypart covers falls to ROTATION — the default, no row needed.
